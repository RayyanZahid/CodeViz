**wsClient.ts — Route graph_delta messages to the activity feed:**

1. In `handleMessage()`, inside the `case 'graph_delta':` block, AFTER the existing `this.queueDelta(msg ...)` call, add:
   ```
   inferenceStore.getState().applyGraphDelta(msg as unknown as GraphDeltaMessage);
   ```
   This sends the same delta message to inferenceStore for feed item generation. The graphStore still gets the full delta for canvas rendering (via queueDelta/flushDeltas). The inferenceStore gets it for feed sentence generation.

   NOTE: The graphStore applyDelta happens on a 500ms batch timer (queueDelta), but applyGraphDelta should process immediately for <3s latency (FEED-01). The feed only needs the addedNodes/removedNodeIds/updatedNodes/addedEdges/removedEdgeIds arrays — it does NOT need them applied to the graph first. So calling applyGraphDelta immediately (not waiting for flushDeltas) is correct.

   IMPORTANT: The `msg` at this point has already been validated by Zod. The cast `as unknown as GraphDeltaMessage` bridges Zod's relaxed types to the strict shared types (same pattern used for InferenceMessage on line ~198).

2. The `import { inferenceStore } from '../store/inferenceStore.js'` is already at the top of wsClient.ts. No new import needed. But do import `GraphDeltaMessage` type:
   - Add `GraphDeltaMessage` to the existing type import from `@archlens/shared/types` on line 4.

**ActivityFeed.tsx — Add live timestamp ticking:**

1. Add `useEffect` import (already imported).

2. Add a `tick` state counter that increments every 10 seconds via setInterval. This forces a re-render so `relativeTime()` recomputes for all visible items. 10-second interval is a reasonable balance — timestamps show "now" for first 30s, then "30s", "1m", "2m" etc. A 10s tick keeps them reasonably fresh without excessive re-renders.

   ```typescript
   const [, setTick] = useState(0);

   useEffect(() => {
     const interval = setInterval(() => setTick((t) => t + 1), 10_000);
     return () => clearInterval(interval);
   }, []);
   ```

   Place this inside the `ActivityFeed` function body, right after the existing `collapsed` and `activityFeed` hooks. The unused first element of `useState` is intentional — we only need the setter to trigger re-renders.

3. No other changes needed in ActivityFeed.tsx. The existing FeedItem already renders `item.iconColor` as the dot color and `relativeTime(item.timestamp)` as the timestamp. Since we're adding the `tick` re-render trigger, timestamps will now update live.
