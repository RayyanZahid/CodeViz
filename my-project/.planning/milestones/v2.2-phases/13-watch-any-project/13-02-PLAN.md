**wsClient.ts** — Handle the `watch_root_changed` message type:

In the `handleMessage` switch statement, add a new case:
```typescript
case 'watch_root_changed': {
  console.log('[WS] Watch root changed to:', msg.directory);
  // 1. Clear graph state — canvas will re-render empty
  graphStore.getState().resetState();
  // 2. Clear inference state — panels will show empty
  inferenceStore.getState().resetState();
  // 3. Update watchRoot in store for UI display
  graphStore.getState().setWatchRoot(msg.directory);
  // 4. Mark as scanning — UI will show loading indicators
  graphStore.getState().setScanning(true);
  // 5. Reset lastQueuedVersion — fresh scan starts from version 0
  this.lastQueuedVersion = 0;
  this.previousVersion = 0;
  // 6. Clear any pending deltas from the old project
  if (this.batchTimer !== null) {
    clearTimeout(this.batchTimer);
    this.batchTimer = null;
  }
  this.pendingDeltas = [];
  break;
}
```

Also: when `initial_state` is received AND `graphStore.getState().scanning` is true, set `scanning` to false (scan has produced results, initial layout is arriving):
```typescript
case 'initial_state': {
  graphStore.getState().applySnapshot(msg as unknown as InitialStateMessage);
  this.lastQueuedVersion = msg.version;
  // If we were scanning after a watch root change, scanning is complete
  if (graphStore.getState().scanning) {
    graphStore.getState().setScanning(false);
  }
  // ... existing change summary logic ...
  break;
}
```

Actually, after a watch_root_changed, the server starts a fresh scan. Delta messages arrive incrementally (not a new initial_state). So scanning should turn off on the FIRST graph_delta received after a watch_root_changed:
```typescript
case 'graph_delta': {
  // Turn off scanning indicator on first delta after watch root change
  if (graphStore.getState().scanning) {
    graphStore.getState().setScanning(false);
  }
  // ... existing gap check and queueDelta logic ...
  break;
}
```

This way the "Scanning..." indicator disappears as soon as the first components start appearing — giving the "live scanning" feel per CONTEXT.md.

Note: The `msg.directory` field is available because we added `watch_root_changed` to the Zod schema in Plan 01. The cast `msg as unknown as WatchRootChangedMessage` follows the existing pattern for other message types if needed, but since Zod parses it, `msg.directory` is directly accessible from the parsed data.

**App.tsx** — Add a `DirectoryBar` component and integrate it into the layout.

**DirectoryBar component** (defined in the same file, following the existing NavButton/PipelineStatusDot pattern of co-located components):

Props: none (reads from graphStore directly via `useGraphStore` hook).

State:
- `inputValue: string` — controlled input, initialized from store's `watchRoot`
- `error: string | null` — inline error message (red text below input)
- `loading: boolean` — true while POST /api/watch is in-flight (disables input)

On mount (useEffect with empty deps):
- Fetch `GET /api/watch` to get initial directory.
- On success, call `graphStore.getState().setWatchRoot(data.directory)` and set `inputValue` to `data.directory`.
- On failure, log warning and leave input empty.

Sync inputValue when `watchRoot` changes externally (e.g., from WsClient receiving watch_root_changed):
- `useEffect(() => { setInputValue(watchRoot); }, [watchRoot])` where `watchRoot` is read via `useGraphStore(s => s.watchRoot)`.

`handleSubmit` function (called on Enter key or button click):
- If `inputValue` is empty or equals current watchRoot, return early.
- Set `loading = true`, `error = null`.
- POST `/api/watch` with body `{ directory: inputValue }`.
- On 200: set `loading = false`. The WsClient will handle the watch_root_changed message and update the store.
- On 400: read error from response JSON, set `error = data.error`, set `loading = false`.
- On network error: set `error = "Failed to connect to server"`, set `loading = false`.
