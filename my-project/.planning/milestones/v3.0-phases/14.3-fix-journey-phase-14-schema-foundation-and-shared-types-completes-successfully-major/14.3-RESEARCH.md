The root problem has two layers: (1) `@playwright/test` is not installed in the project at all — it is absent from `package.json` devDependencies and not resolvable from `node_modules`; (2) three files in `.auto-gsd/journey-tests/` have been overwritten by auto-gsd with regressed versions that strip out the critical `webServer` block, ESM `__dirname`/`projectRoot` setup, pnpm command usage, and the dist verification test.

The installed playwright/playwright-core in the npx cache does NOT include `@playwright/test` (they are separate packages — `playwright` is the all-in-one browser automation library, `@playwright/test` is the test runner framework). The project uses pnpm workspaces (packages/client, packages/server, packages/shared) and the workspace configuration does NOT include `.auto-gsd/journey-tests` — journey tests are outside the workspace. This makes creating a standalone `package.json` in `.auto-gsd/journey-tests/` the cleanest approach.

The user decision explicitly switched from the Phase 14.1 approach (adding `@playwright/test` to root `package.json`) to a standalone `package.json` in `.auto-gsd/journey-tests/`. This is motivated by regression prevention: if auto-gsd regenerates the root `package.json`, the dependency survives in the journey-tests directory's own manifest. The scope also covers Phase 15 journey tests since `journey-phase-15.spec.ts` (untracked, currently a stub) shares the same broken infrastructure.

**Primary recommendation:** Create `.auto-gsd/journey-tests/package.json` with `@playwright/test`, install with `npm install` inside that directory, restore the three regressed files from git HEAD, and then rewrite the Phase 15 journey tests with real API endpoint assertions matching the Phase 14 pattern.

## Architecture Patterns

### Standalone package.json in journey-tests (Locked Decision)

The user decided against adding `@playwright/test` to root `package.json` (Phase 14.1's approach) because auto-gsd regenerates root `package.json`. A standalone `package.json` in `.auto-gsd/journey-tests/` is isolated from workspace tooling and auto-gsd regeneration.

**Key constraint:** The pnpm workspace is defined as `packages/*` only — `.auto-gsd/journey-tests` is NOT in the workspace. This means:
- Installing dependencies in `.auto-gsd/journey-tests/` uses plain `npm install` (not pnpm workspace commands)
- The directory will have its own `node_modules/@playwright/test`
- `playwright.config.ts` imports resolve to `.auto-gsd/journey-tests/node_modules/@playwright/test`

### Recommended package.json structure

```json
{
  "name": "archlens-journey-tests",
  "version": "1.0.0",
  "private": true,
  "type": "module",
  "scripts": {
    "test": "playwright test"
  },
  "devDependencies": {
    "@playwright/test": "^1.50.0"
  }
}
```

**Why `"type": "module"`:** The spec files use `import` syntax and the config uses `fileURLToPath` — ESM is required. The project's root `package.json` already has `"type": "module"`.

**Why `^1.50.0` and not the latest:** Phase 14.1 plan specified `^1.50.0`. Current available version is 1.58.2. Either works; planner can choose exact version.

### Phase 15 journey tests: placeholder to real tests

The current `.auto-gsd/journey-tests/journey-phase-15.spec.ts` (untracked, not in git) is a stub with 4 tests that all do `page.goto(BASE_URL + '/')` and check `body.length`. These need to be rewritten to call the real Phase 15 API endpoints.

**Phase 15 journey test pattern (Truth 1 — timeline):**
```typescript
test('GET /api/timeline returns snapshot metadata after file changes', async ({ page }) => {
  test.setTimeout(60000);
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'archlens-ph15-'));
  try {
    for (let i = 0; i < 6; i++) {
      fs.writeFileSync(path.join(tmpDir, `mod${i}.js`), `export const v${i} = ${i};\n`);

      await page.waitForTimeout(1000);
      const res = await page.request.get(`${SERVER_URL}/api/timeline`);

    expect(snapshots.length).toBeGreaterThan(0);
    const first = snapshots[0] as Record<string, unknown>;
    expect(typeof first.id).toBe('number');
    expect(typeof first.sequenceNumber).toBe('number');
    expect(typeof first.timestamp).toBe('number');  // epoch ms, not ISO string
  } finally {
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| `__dirname` in ESM | Custom CWD detection | `path.dirname(fileURLToPath(import.meta.url))` | Standard Node.js ESM pattern; CWD is process-dependent and breaks when called from different directories |
| Package resolution for `@playwright/test` | Symlinks or manual copies | `npm install` in `.auto-gsd/journey-tests/` | Node module resolution requires the package in `node_modules/` on the resolution path |
| Browser binary installation | Manual binary placement | `npx playwright install chromium` or `node_modules/.bin/playwright install chromium` | Playwright manages browser binary paths and versions internally |

**Key insight:** The `ERR_MODULE_NOT_FOUND` for `@playwright/test` cannot be fixed by using `npx playwright test` — the `npx playwright` command invokes the `playwright` CLI package which internally requires `@playwright/test` to be installed in the project. The package must exist in `node_modules` on a path that Node's `import` resolution can find from the spec files' location.

## Common Pitfalls

### Pitfall 1: Installing @playwright/test at root instead of in journey-tests/

**What goes wrong:** Adding `@playwright/test` to root `package.json` devDependencies (Phase 14.1 pattern) works initially but auto-gsd regenerates root `package.json`, stripping the dependency. The same regression happens again.

**Why it happens:** Root `package.json` is auto-gsd managed; `.auto-gsd/journey-tests/package.json` is not.

**How to avoid:** Create `package.json` inside `.auto-gsd/journey-tests/` (locked decision). This directory is not auto-gsd managed.

**Warning signs:** After next auto-gsd run, `@playwright/test` missing from root `package.json` again.

### Pitfall 2: webServer only starts Vite (port 5173), but Phase 14 tests hit Fastify (port 3100)

**What goes wrong:** webServer `url` is checked against port 5173 (Vite) for readiness, but all phase-14 tests use `SERVER_URL = 'http://localhost:3100'` (Fastify). If pnpm dev doesn't start Fastify, tests will ECONNREFUSED port 3100.

**Why it happens:** The `pnpm dev` command starts BOTH Vite and Fastify concurrently (see root `package.json` scripts). Playwright's webServer block only checks port 5173 for readiness but relies on `pnpm dev` having started both processes.

**How to avoid:** The `pnpm dev` script is `pnpm build:workers && concurrently -n server,client -c cyan,yellow "pnpm --filter @archlens/server dev" "pnpm --filter @archlens/client dev"` — both server (Fastify:3100) and client (Vite:5173) are started. The webServer readiness check at 5173 is sufficient because both processes start together. The 120s timeout accommodates the TypeScript compilation delay.

**Warning signs:** Tests receive ECONNREFUSED on port 3100 even though Playwright reports the server started. Check if Fastify crashed with an error (SQLite, port conflict, etc.).

### Pitfall 3: journey-phase-15 tests still use page.goto instead of page.request

**What goes wrong:** The current `journey-phase-15.spec.ts` on disk does `page.goto(BASE_URL + '/')` against Vite port 5173 — this doesn't test Phase 15 server functionality at all.

**Why it happens:** The file was auto-generated as a stub, same as the old Phase 14 tests before Phase 14.2 fixed them.

**How to avoid:** Rewrite all 4 Phase 15 tests using `page.request` against `SERVER_URL` (port 3100), following the Phase 14 pattern exactly. The scope of Phase 14.3 explicitly includes Phase 15 journey test rewrite.

### Pitfall 4: Working tree regressions silently overwrite git-tracked files

**What goes wrong:** After this phase, auto-gsd may overwrite `playwright.config.ts` and the spec files again.

**Why it happens:** auto-gsd generates test files based on phase specs without respecting existing content.

**How to avoid:** The locked decision is: restore from git is the fix. The committed state is the source of truth. No special tooling is needed — `git checkout HEAD -- <files>` restores them. Planner should include a note in the PLAN about this recovery strategy.

### Pitfall 5: @playwright/test version mismatch with installed playwright binary

**What goes wrong:** If `@playwright/test@1.50.0` is installed but the installed browser binaries are from playwright@1.58.2 (the npx-cached version), the browser binary may not match.

**Why it happens:** Playwright tightly couples the test runner version to browser binary versions.

**How to avoid:** After installing `@playwright/test`, run browser install using the locally-installed playwright binary: `./node_modules/.bin/playwright install chromium` from within `.auto-gsd/journey-tests/`. This ensures version consistency.

```typescript
  test.setTimeout(60000);

    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```
