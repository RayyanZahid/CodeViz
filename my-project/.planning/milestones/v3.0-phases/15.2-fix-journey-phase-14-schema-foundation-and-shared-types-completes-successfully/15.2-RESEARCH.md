### Pattern 2: journey-phase-14.spec.ts (New content per CONTEXT.md decisions)

The CONTEXT.md decisions differ from the HEAD commit in the following ways:
- HEAD delta threshold test: writes 7 files, then polls up to 30 seconds for snapshots
- CONTEXT.md decision: writes 15-20 files, waits **5 seconds fixed** (not polling), then checks snapshot count
- CONTEXT.md: include descriptive custom failure messages

**Key behavioral note:** The CONTEXT.md says "Use /api/timeline to check snapshot count after changes settle" and "Wait ~5 seconds (fixed wait) for the SnapshotManager's 3s debounce to settle." However, the timeline endpoint returns snapshots for the current session. After a `POST /api/watch` to a temp directory, the new SnapshotManager session has ID = new UUID. The initial scan may produce an "initial snapshot" (from `captureInitialSnapshot()` called 2s after pipeline start). After the initial snapshot is taken, writing 15-20 additional files will all be debounced into 1 more snapshot. Total: 1 initial + 1 after burst = 2 snapshots for 15-20 files.

**Wait timing analysis:**
- Pipeline starts when `/api/watch` returns 200
- `captureInitialSnapshot()` fires 2s after pipeline start
- Writing 15-20 files triggers debounce
- Debounce fires 3s after last delta
- Total from `/api/watch` response to last snapshot: ~2s (initial) + file write time + 3s (debounce) ≈ 5-6s

The CONTEXT.md's 5s fixed wait applies to after the file writes are complete. The test should:
1. POST /api/watch, wait for 200
2. Write 15-20 files to temp dir
3. Wait 5 seconds (fixed)
4. GET /api/timeline, assert count < file count

**IMPORTANT TIMING EDGE CASE:** A 5s fixed wait may not be sufficient if pipeline scan is slow. The prior phase (14.2/15.1) used a polling loop for robustness. The CONTEXT.md says "fixed wait" so honor that, but add a note in the test about potential flakiness on slow machines.

**Shared types endpoint behavior:** The current diagnostic endpoint returns:
```json
{
  "ok": true,
  "compilation": "server_running_proves_compilation",
  "hasIntentCategory": true
}
```
The CONTEXT.md says to "verify specific type names (SnapshotMeta, IntentSession) are importable/exist at runtime." These are TypeScript interfaces — they are type-erased at runtime. The check must be on the runtime module exports, not class/value names. The current endpoint does NOT return `SnapshotMeta` or `IntentSession` fields. The CONTEXT.md says "call /api/debug/shared-types endpoint AND also verify specific type names... are importable/exist at runtime."

Two options:
1. Update the diagnostic endpoint to check for `SnapshotMeta` and `IntentSession` — but these are interfaces (erased at runtime), so the endpoint can't find them by name in module exports
2. The test checks `ok: true` from the endpoint (server running = compiled) AND interprets "exist at runtime" as: the `IntentCategory` const object is accessible (it's the only runtime value export from timeline.ts)

**Resolution:** The most pragmatic interpretation that satisfies the spirit: the diagnostic endpoint already checks `hasIntentCategory: true` which proves the timeline module compiled and exported. The test should assert `body.ok === true` (proves compilation) and `body.hasIntentCategory === true` (proves runtime access to shared types module). This honors the "verify specific type names... exist at runtime" requirement via the const object that represents the intent type taxonomy. The test description can include language about "runtime access to IntentCategory (the const companion to the IntentSession type)" to make the intent clear.

### Pattern 3: Delta Threshold Test Structure (15-20 files, fixed 5s wait)

```typescript
// Source: CONTEXT.md locked decisions + codebase investigation
test('snapshot writes triggered only at delta threshold, not every event', async ({ page }) => {
  test.setTimeout(60000);

  // Use GET /api/watch to discover current watch root (for context)
  const watchRes = await page.request.get(`${SERVER_URL}/api/watch`);
  expect(watchRes.status(), 'GET /api/watch should return 200').toBe(200);

  // Create temp directory and write 15-20 valid JS files
  const { mkdtempSync, writeFileSync } = await import('fs');
  const { join } = await import('path');
  const { tmpdir } = await import('os');
  const tmpDir = mkdtempSync(join(tmpdir(), 'archlens-diag-'));
  const FILE_COUNT = 18;  // Within the 15-20 range

  for (let i = 0; i < FILE_COUNT; i++) {
    writeFileSync(
      join(tmpDir, `module${i}.js`),
      `export const value${i} = ${i};\nexport function fn${i}() { return ${i}; }\n`
    );
  }

  // Switch watcher to temp directory (files already exist for initial scan)
  const switchRes = await page.request.post(`${SERVER_URL}/api/watch`, {
    data: { directory: tmpDir },
  });
  expect(switchRes.status(), 'POST /api/watch should succeed').toBe(200);

  // Fixed 5s wait for SnapshotManager 3s debounce to settle after pipeline scan
  await page.waitForTimeout(5000);

  // Check snapshot count via /api/timeline
  const timelineRes = await page.request.get(`${SERVER_URL}/api/timeline`);
  expect(timelineRes.status(), 'GET /api/timeline should return 200').toBe(200);
  const snapshots = await timelineRes.json() as unknown[];

  // Assert: fewer snapshots than file events proves delta throttling is active
  expect(
    snapshots.length,
    `Expected snapshot count < file count but got ${snapshots.length} snapshots for ${FILE_COUNT} file events`
  ).toBeLessThan(FILE_COUNT);

  // At least 1 snapshot must exist (proves the pipeline did run)
  expect(
    snapshots.length,
    `Expected at least 1 snapshot after ${FILE_COUNT} file events but got 0`
  ).toBeGreaterThan(0);
});
```

**Why write files BEFORE the `/api/watch` switch:** The CONTEXT.md says to "create temp .ts files there for file change triggers." Writing files before the switch means the initial scan picks them up as a burst. The SnapshotManager will see N added nodes (structural events), schedule a debounced snapshot, and the initial snapshot + debounce snapshot account for much fewer than 18 writes. After the 5s wait, timeline should have 1-2 snapshots, not 18.

**Alternative: write AFTER switch:** If files are written after the switch, some may arrive during scan and some may arrive afterward, creating a more complex race condition. Writing before the switch is cleaner and more deterministic.

## Common Pitfalls

### Pitfall 1: Restoring journey-phase-14.spec.ts from Git HEAD (Wrong Approach)

**What goes wrong:** Developer runs `git checkout HEAD -- .auto-gsd/journey-tests/journey-phase-14.spec.ts` and stops. The tests pass based on the old decisions (7 files, polling), but the new CONTEXT.md decisions (15-20 files, fixed 5s wait) are not honored.

**Why it happens:** This is a different phase from 14.3 and 15.1 which were pure "restore from git" phases. Phase 15.2 has new decisions in CONTEXT.md that supersede what's in HEAD for `journey-phase-14.spec.ts`.

**How to avoid:** Restore the three infrastructure files from HEAD. Write `journey-phase-14.spec.ts` fresh per CONTEXT.md locked decisions.

**Warning signs:** Test uses polling loop with 30s timeout; writes only 7 files.

### Pitfall 2: Fixed 5s Wait Flakiness on Slow Machines

**What goes wrong:** The 5s fixed wait is not sufficient on a slow machine. SnapshotManager debounce is 3s, but pipeline scan time depends on disk I/O. If the scan takes >2s, the debounce hasn't fired before the 5s wait elapses. Timeline returns 0 snapshots.

**Why it happens:** The pipeline parses files with tree-sitter workers (ParserPool). Cold start of workers can add 1-2 additional seconds.

**How to avoid:** The CONTEXT.md locked decision says "fixed wait" so we honor it. However, we can add an additional assertion that retries once: if 0 snapshots after 5s, wait another 3s and retry. But strictly, per CONTEXT.md: use 5s. If flaky in CI, the timeout can be increased without changing the test structure.

**Note:** Phase 15.1 UAT showed all 4 Phase 14 tests passed, which used the 7-file/30s-polling approach. The 15-20 files approach is less likely to have timing issues because the initial scan captures all files at once.

**Warning signs:** `snapshots.length === 0` assertion failure; server log shows no snapshot captured.

### Pitfall 3: Port 5173 vs 3100 Confusion

**What goes wrong:** Tests use `BASE_URL = http://localhost:5173` instead of `SERVER_URL = http://localhost:3100` for API calls. Diagnostic endpoints don't exist on the Vite port.

**Why it happens:** Auto-gsd regenerated placeholder tests use `BASE_URL` (Vite). Real tests need `SERVER_URL` (Fastify).

**How to avoid:** Define `const SERVER_URL = process.env.SERVER_URL || 'http://localhost:3100'` at the top of `journey-phase-14.spec.ts`.

**Warning signs:** 404 responses from `page.request.get()`.

### Pitfall 4: NODE_ENV Not Set in Playwright Process vs webServer Process

**What goes wrong:** Setting `NODE_ENV=development` in `playwright.config.ts`'s `webServer.env` only affects the webServer child process (pnpm dev). The Playwright test process itself may have a different NODE_ENV. This is fine because the server process is what needs NODE_ENV to enable diagnostic endpoints.

**Why it's not a problem:** The `index.ts` check is `if (process.env.NODE_ENV !== 'production')` — this is evaluated in the server process, not the test process. The `webServer.env` configuration correctly sets it for the server.

**How to handle:** If `reuseExistingServer: true` and the server was already started WITHOUT `NODE_ENV=development` explicitly, the diagnostic endpoints should still be available (NODE_ENV is not 'production' by default). This is a belt-and-suspenders guard.

### Pitfall 5: journey-phase-15.spec.ts validCategories — Lowercase Snake_case Required

**What goes wrong:** `journey-phase-15.spec.ts` HEAD version has a `validCategories` array. If it contains uppercase values (FEATURE_BUILDING), the test fails when comparing against the IntentAnalyzer's actual string output (which uses lowercase 'feature_building').

**Why it happens:** Phase 15.1 discovered and fixed this: the IntentCategory const uses lowercase snake_case values. HEAD already has the fix (`validCategories = ['feature_building', 'bug_fixing', ...]`). Restoring from HEAD should include the correct lowercase values.

**Warning signs:** `validCategories.includes(session.category)` is false even when session has a valid category.

## Open Questions

1. **Will a fixed 5s wait be sufficient for the delta threshold test?**
   - What we know: SnapshotManager debounce is 3s. Pipeline startup + scan is ~2s cold. 5s total = 2s (scan) + 3s (debounce) = right at the boundary.
   - What's unclear: ParserPool worker initialization time on this specific machine.
   - Recommendation: Use 5s per CONTEXT.md. If flaky, the fix is to increase to 7s or add a minimal retry (but stay within the "fixed wait" spirit). Phase 15.1 showed all 4 Phase 14 tests passed — the server is warmed up during test runs, so 5s should be fine when reuseExistingServer:true allows the server to start before tests run.

2. **Should `journey-phase-14.spec.ts` use `test.describe.configure({ mode: 'serial' })` for test ordering?**
   - What we know: The delta threshold test POSTs to `/api/watch` to change the watch root globally. This affects all subsequent tests in the session.
   - What's unclear: Whether tests run in order within the same describe block by default.
   - Recommendation: Playwright runs tests within a `describe` in order by default. Making the delta threshold test LAST ensures the three CRUD/types tests run against the original watch root. No explicit `serial` mode needed if order is preserved.

3. **Does the `/api/debug/shared-types` endpoint need to be updated to return `SnapshotMeta`/`IntentSession` field names?**
   - What we know: These are TypeScript interfaces — erased at runtime. The endpoint cannot expose them as runtime names.
   - What's unclear: Whether the CONTEXT.md expectation is satisfiable as stated.
   - Recommendation: The `hasIntentCategory: true` check adequately satisfies the spirit of "verify specific type names exist at runtime" — IntentCategory is the runtime const companion to the IntentSession type from `timeline.ts`. The test message can explicitly state this is the runtime proof of the shared types module being accessible. No endpoint changes needed.
