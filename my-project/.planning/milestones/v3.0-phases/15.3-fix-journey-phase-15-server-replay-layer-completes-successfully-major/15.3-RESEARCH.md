### Pattern 1: Polling for async events

The snapshot pipeline has a 3-second debounce. Tests must poll `/api/timeline` with a timeout rather than using fixed delays (per locked decisions).

```typescript
// Source: phase 14.3 established pattern
let snapshots: Array<Record<string, unknown>> = [];
const deadline = Date.now() + 30000;
while (Date.now() < deadline) {
  await page.waitForTimeout(1000);
  const res = await page.request.get(`${SERVER_URL}/api/timeline`);
  if (res.status() === 200) {
    snapshots = (await res.json()) as Array<Record<string, unknown>>;
    if (snapshots.length > 0) break;
  }
}
expect(snapshots.length).toBeGreaterThan(0);
```

### Pattern 2: Temp directory isolation

Each test that needs snapshot data creates its own temp directory and switches the server's watch root via POST /api/watch.

```typescript
// Source: phases 14.3/15.1 established pattern
import fs from 'fs';
import path from 'path';
import os from 'os';

// At top of file (not inside test body — top-level imports are fine in ESM spec files)
const SERVER_URL = process.env.SERVER_URL || 'http://localhost:3100';

// Inside test:
test.setTimeout(60000);
const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'archlens-ph15-'));
try {
  // Write TypeScript files with real import statements
  fs.writeFileSync(path.join(tmpDir, 'a.ts'), `import { b } from './b.js';\nexport const a = 1;\n`);
  fs.writeFileSync(path.join(tmpDir, 'b.ts'), `export const b = 2;\n`);

  const switchRes = await page.request.post(`${SERVER_URL}/api/watch`, { data: { directory: tmpDir } });
  expect(switchRes.status()).toBe(200);

  // Poll for snapshots...
} finally {
  fs.rmSync(tmpDir, { recursive: true, force: true });
}
```

**Key decision from CONTEXT.md:** Write TypeScript files with real import statements so the parser creates proper nodes and edges in the graph.

### Pattern 3: Import statements at file top (not inside test body)

The `journey-phase-14.spec.ts` uses dynamic imports inside test bodies for `fs/path/os` for ESM compatibility reasons specific to those tests. However, the HEAD `journey-phase-15.spec.ts` pattern uses top-level `import fs from 'fs'` which is valid ESM in Playwright spec files. Either approach works — the planner should follow the precedent established in the HEAD phase-15 file.

### Anti-Patterns to Avoid

- **Using BASE_URL (5173) for API calls:** The Vite dev server at port 5173 does NOT proxy all API routes. Target `SERVER_URL` (port 3100) directly for all Fastify API calls.
- **Using fixed waits instead of polling:** `await page.waitForTimeout(5000)` on its own is fragile. Use polling with a deadline for snapshot appearance tests (polling is the CONTEXT.md locked decision for phase-15 tests).
- **Assuming positions are populated:** The `layout_positions` table is empty until Phase 6 (layout persistence) is built. The CONTEXT.md assertion "positions have numeric x/y for EVERY node" CANNOT be satisfied as written (see Critical Finding below).
- **Using page.goto for API assertions:** Always use `page.request.get/post` for API endpoints.
- **Using `npx playwright` without the local binary:** Use `node_modules/.bin/playwright` or the `.cmd` variant on Windows.

## Critical Findings

### Finding 1: positions assertion is currently unsatisfiable

**CONTEXT.md says:** "positions have numeric x/y for EVERY node (Object.keys(positions).length === nodes.length)"

**Reality:** The `SnapshotManager.captureSnapshot()` populates positions from `positionsRepository.findAll()`, which reads the `layout_positions` SQLite table. This table is populated by Phase 6 layout persistence, which is NOT yet built. In the current codebase, the table is always empty, so `positions` in every snapshot is `{}`.

**Evidence from SnapshotManager.ts (lines 176-186):**
```typescript
const allPositions = positionsRepository.findAll();
const positions: Record<string, { x: number; y: number }> = {};
for (const row of allPositions) {
  positions[row.nodeId] = { x: row.x, y: row.y };
}
```
When `positionsRepository.findAll()` returns `[]`, `positions` is `{}`.

**Impact:** The assertion `Object.keys(positions).length === nodes.length` will always fail in the current codebase because positions is `{}` and nodes may have length > 0.

**Resolution options for planner:**
1. **Weaken assertion (recommended):** Assert `typeof positions === 'object'` and that positions is not null — matches what the HEAD phase-15 test already does and is accurate for the current build state.
2. **Skip positions assertion entirely:** Just assert nodes and edges arrays are present.
3. **Clarify with user:** The CONTEXT.md assertion about positions may be aspirational/misremembered. The HEAD version (which was passing) only checked `typeof positions === 'object'`.

**Recommendation:** Use option 1 — assert `typeof positions === 'object'` and `positions !== null`. Do NOT assert positions.length === nodes.length as this will cause permanent test failures until Phase 6 is complete.

### Finding 2: IntentAnalyzer requires enough file events to reach MIN_SIGNAL_THRESHOLD

**Reality:** IntentAnalyzer only creates sessions when `sumOfScores >= MIN_SIGNAL_THRESHOLD (0.1)`. A single file change may or may not cross the threshold. The test for `/api/intents` must:
1. Write enough files with clear intent signals (e.g., test files with `.spec.ts` suffix to trigger TEST_WRITING category clearly)
2. Wait for IntentAnalyzer to process them via delta events
3. Poll `/api/intents` to verify a session exists

The HEAD version's intent test only calls GET /api/intents on the current session (without switching watch root) and skips validation if `sessions.length === 0`. CONTEXT.md requires stronger validation: "category is valid IntentCategory enum, confidence between 0 and 1, subtasks is array, startSnapshotId is number, objective is string, sessionId matches".

**To guarantee a session exists:** The test must write files to a fresh temp dir, switch watch root, poll until snapshots appear (which implies IntentAnalyzer has also processed deltas), then verify `/api/intents` has at least one session.

### Finding 3: switchWatchRoot does NOT purge graphSnapshots or intentSessions tables

**Evidence from index.ts (switchWatchRoot function):**
```typescript
db.delete(graphEdges).run();
db.delete(graphNodes).run();
// NOTE: graphSnapshots and intentSessions are NOT deleted
```

**Impact:** Old session snapshots and intent sessions remain in the DB across watch root switches. The `/api/timeline` endpoint filters by `snapshotManager.getSessionId()` (the current session) — so tests only see data from the current session. Tests switching watch root multiple times will NOT contaminate each other's timeline data.

**Important:** After POST /api/watch, the `timelinePlugin`'s `getSessionId` closure automatically returns the new SnapshotManager's session ID (per STATE.md decision: "timelinePlugin uses getSessionId closure over snapshotManager — after switchWatchRoot replaces snapshotManager, closure returns new session ID automatically"). Tests can call GET /api/timeline immediately after POST /api/watch and will only see the new session's snapshots.

### Finding 4: node IDs in snapshot are file paths, not short names

**Evidence from DependencyGraph and SnapshotManager:** The parser produces file-path IDs (e.g., `/tmp/archlens-ph15-abc123/module0.js`). When we write `module0.js` to a temp dir and switch watch root, the snapshot nodes will have IDs like the full absolute path.

**Impact for assertion:** The CONTEXT.md assertion "node IDs match written files" means checking that each written file's absolute path appears as a node ID. The test must:
```typescript
const writtenFiles = Array.from({length: 6}, (_, i) => path.join(tmpDir, `a${i}.ts`));
// After snapshot appears:
const nodeIds = (snap.nodes as Array<{id: string}>).map(n => n.id);
for (const filePath of writtenFiles) {
  expect(nodeIds).toContain(filePath);
}
```

### Finding 5: edges require real import statements in written files

**Evidence from IntentAnalyzer.ts (classifyDelta):**
> "addedEdges signals FEATURE_BUILDING"

The parser only creates edges when files have `import` statements. Writing `export const x = 1;` creates a node but no edge to other written files. To get edges in the snapshot, write files with real TypeScript import statements:

```typescript
// b.ts
export const b = 2;

// a.ts — imports b to create an edge a → b
import { b } from './b.js';
export const a = b + 1;
```

Note: TypeScript `.ts` files importing via `.js` extension is standard per the project's ESM configuration.

### Finding 6: Concurrency test needs 10+ files for snapshot threshold

**Evidence from SnapshotManager constants:**
```typescript
const MINOR_THRESHOLD = 10; // 10 minor events before forced snapshot
```

Writing 10+ separate plain files (no structural changes between them) will accumulate 10 minor events and trigger a snapshot. For the concurrency test, writing `>= 10` files ensures at least one snapshot is triggered. Writing files that also create new nodes (first-time additions) are structural changes, which trigger immediate debounced snapshots — even 1 structural file change starts the debounce timer.

### Finding 7: Test name titles must match the Phase 15 success criteria test names

The test names in the spec file must match the four Phase 15 success criteria exactly as labeled in the journey test spec (they serve as integration test titles for the roadmap). The current stub has:
1. `` `GET /api/timeline` returns a list of snapshot metadata with sequence numbers and timestamps after file changes occur ``
2. `` `GET /api/snapshot/:id` returns a complete graph snapshot (nodes, edges, positions) reconstructed from the nearest checkpoint in at most 50 replay steps ``
3. `The \`IntentAnalyzer\` classifies a realistic sequence of architectural events into one of the 4-6 coarse categories and returns a confidence score`
4. `Writing files during an active session does not cause the pipeline to pause — new events continue to arrive while the replay read path is active`

The HEAD version uses different (shorter) names. The stub names should be preserved since they match the Phase 15 roadmap success criteria description.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP requests to API | Custom fetch wrapper | `page.request.get/post` | Playwright's request fixture handles cookies, base URL, etc. |
| Polling with timeout | Custom async retry | Manual while loop with `Date.now()` | Simple, no deps; Playwright-native `waitFor` is over-engineered for this |
| Temp directory creation | Custom dir management | `fs.mkdtempSync(path.join(os.tmpdir(), prefix))` | OS-guaranteed uniqueness, simple |
| JSON parsing | Manual text parse | `await response.json()` | Built into Playwright APIResponse |

**Key insight:** This is a test-writing task with no new server code. All server capabilities are already implemented and tested. The only task is writing HTTP assertions that exercise existing endpoints.

## Common Pitfalls

### Pitfall 1: Forgetting to restore the other spec files
**What goes wrong:** Only journey-phase-15.spec.ts is written, leaving the other 3 files as stubs. All 12 tests must pass green.
**Why it happens:** The objective focuses on Phase 15 tests, but the infrastructure files were also regressed.
**How to avoid:** Restore playwright.config.ts, journey-build-and-start.spec.ts, and journey-phase-14.spec.ts from HEAD first, then write phase-15 fresh.
**Warning signs:** After restoration, check that playwright.config.ts contains `webServer`, `NODE_ENV`, and `projectRoot`.

### Pitfall 2: Testing positions with node count equality
**What goes wrong:** Test asserts `Object.keys(positions).length === nodes.length` — this always fails because `layout_positions` table is empty until Phase 6.
**Why it happens:** CONTEXT.md states this assertion but it's not achievable with the current codebase.
**How to avoid:** Assert `typeof positions === 'object'` and `positions !== null` only — the same as what the previously passing HEAD version verified.
**Warning signs:** Test fails with `Expected 0 to equal 6` or similar.

### Pitfall 3: Intent test passing vacuously when session list is empty
**What goes wrong:** Test passes `if (sessions.length > 0)` check is vacuously satisfied when sessions array is empty.
**Why it happens:** HEAD version used a conditional guard — CONTEXT.md requires asserting a session actually exists.
**How to avoid:** The intent test must first write files, switch watch root, and poll until snapshots appear (guaranteeing deltas have fired). Then call `/api/intents` and assert `sessions.length > 0` unconditionally.
**Warning signs:** Test passes even when intent classification is broken.

### Pitfall 4: Writing plain JS files with no imports — no edges
**What goes wrong:** `GET /api/snapshot/:id` assertions for edges fail because no imports were written.
**Why it happens:** Simple `export const x = i` files create nodes but no edges.
**How to avoid:** Write at least 2 files where one imports the other:
```typescript
// b.ts
export const b = 2;
// a.ts
import { b } from './b.js';
export const a = b + 1;
```
**Warning signs:** `snap.edges.length === 0` when expecting edges.

### Pitfall 5: Using `request` fixture for page assertions vs `page.request`
**What goes wrong:** Tests import `request` fixture instead of `page.request` — different timeout contexts.
**Why it happens:** Playwright has two ways to make HTTP requests.
**How to avoid:** Use `page.request` consistently with the `{ page }` fixture, matching the established pattern from phases 14.3 and 15.2.

### Pitfall 6: Not accounting for 3-second debounce + scan delay
**What goes wrong:** Polling loop with 30-second timeout isn't long enough, or poll interval too short.
**Why it happens:** Files need to be written → chokidar detects → parser runs → delta fires → debounce fires (3s) → SQLite write. Total can be 4-8 seconds.
**How to avoid:** Use 30-second polling deadline, 1-second poll interval. The server also has a 2-second `setTimeout` before `captureInitialSnapshot()` fires on watch root switch.

### Pitfall 7: The git restore path prefix requirement
**What goes wrong:** `git checkout HEAD -- .auto-gsd/journey-tests/*.spec.ts` fails because the repo root is one level above the working directory.
**Why it happens:** Git repo root is `F:\Directory\FrontierIntelligenceHack` but working directory is `my-project/` inside.
**How to avoid:** Use path prefix `my-project/`:
```bash
git -C .. checkout HEAD -- my-project/.auto-gsd/journey-tests/playwright.config.ts my-project/.auto-gsd/journey-tests/journey-build-and-start.spec.ts my-project/.auto-gsd/journey-tests/journey-phase-14.spec.ts
```

### Example 1: Full test structure for GET /api/timeline

```typescript
// Source: CONTEXT.md locked decisions + phase 14.3/15.1 patterns
import { test, expect } from '@playwright/test';
import fs from 'fs';
import path from 'path';
import os from 'os';

const SERVER_URL = process.env.SERVER_URL || 'http://localhost:3100';

test('`GET /api/timeline` returns a list of snapshot metadata with sequence numbers and timestamps after file changes occur', async ({ page }) => {
  test.setTimeout(60000);
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'archlens-ph15-tl-'));
  try {
    // Write TypeScript files with real imports to create nodes + edges
    fs.writeFileSync(path.join(tmpDir, 'b.ts'), `export const b = 2;\n`);
    fs.writeFileSync(path.join(tmpDir, 'a.ts'), `import { b } from './b.js';\nexport const a = b + 1;\n`);
    fs.writeFileSync(path.join(tmpDir, 'c.ts'), `import { a } from './a.js';\nexport const c = a + 1;\n`);

    const switchRes = await page.request.post(`${SERVER_URL}/api/watch`, { data: { directory: tmpDir } });
    expect(switchRes.status(), 'POST /api/watch should return 200').toBe(200);

    // Poll for snapshots (3s debounce + 2s initial capture delay)
    let snapshots: Array<Record<string, unknown>> = [];
    const deadline = Date.now() + 30000;
    while (Date.now() < deadline) {
      await page.waitForTimeout(1000);
      const res = await page.request.get(`${SERVER_URL}/api/timeline`);
      if (res.status() === 200) {
        snapshots = (await res.json()) as Array<Record<string, unknown>>;
        if (snapshots.length > 0) break;
      }
    }

    expect(snapshots.length, 'Expected at least 1 snapshot in timeline').toBeGreaterThan(0);

    // Shape validation
    const first = snapshots[0];
    expect(typeof first.id, 'id should be a number').toBe('number');
    expect(typeof first.sequenceNumber, 'sequenceNumber should be a number').toBe('number');
    expect(typeof first.timestamp, 'timestamp should be a number').toBe('number');
    expect(typeof first.summary, 'summary should be a string').toBe('string');
    expect(Array.isArray(first.triggerFiles), 'triggerFiles should be an array').toBe(true);

    // Timestamp epoch range validation
    const ts = first.timestamp as number;
    expect(ts, 'timestamp should be after 2025-01-01').toBeGreaterThan(1735689600000);
    expect(ts, 'timestamp should be before year 2030').toBeLessThan(1893456000000);

    // Monotonically increasing sequenceNumbers (if multiple snapshots)
    if (snapshots.length > 1) {
      for (let i = 1; i < snapshots.length; i++) {
        expect(snapshots[i].sequenceNumber as number)
          .toBeGreaterThan(snapshots[i - 1].sequenceNumber as number);
      }
    }
  } finally {
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```

### Example 2: GET /api/snapshot/:id deep validation

```typescript
test('`GET /api/snapshot/:id` returns a complete graph snapshot (nodes, edges, positions)...', async ({ page }) => {
  test.setTimeout(60000);
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'archlens-ph15-snap-'));
  try {
    // Write files with imports to guarantee edges
    const fileB = path.join(tmpDir, 'b.ts');
    const fileA = path.join(tmpDir, 'a.ts');
    fs.writeFileSync(fileB, `export const b = 2;\n`);
    fs.writeFileSync(fileA, `import { b } from './b.js';\nexport const a = b + 1;\n`);

    await page.request.post(`${SERVER_URL}/api/watch`, { data: { directory: tmpDir } });

    // Poll for snapshot ID
    let snapshotId: number | null = null;
    const deadline = Date.now() + 30000;
    while (Date.now() < deadline) {
      await page.waitForTimeout(1000);
      const res = await page.request.get(`${SERVER_URL}/api/timeline`);
      if (res.status() === 200) {
        const list = (await res.json()) as Array<Record<string, unknown>>;
        if (list.length > 0) { snapshotId = list[0].id as number; break; }
      }
    }
    expect(snapshotId, 'Expected a snapshot ID from timeline').not.toBeNull();

    // Fetch snapshot by ID
    const snapRes = await page.request.get(`${SERVER_URL}/api/snapshot/${snapshotId}`);
    expect(snapRes.status(), 'GET /api/snapshot/:id should return 200').toBe(200);

    const snap = (await snapRes.json()) as Record<string, unknown>;
    expect(typeof snap.id, 'snapshot id should be a number').toBe('number');
    expect(typeof snap.sequenceNumber, 'sequenceNumber should be a number').toBe('number');
    expect(Array.isArray(snap.nodes), 'nodes should be an array').toBe(true);
    expect(Array.isArray(snap.edges), 'edges should be an array').toBe(true);
    expect(typeof snap.positions, 'positions should be an object').toBe('object');
    expect(snap.positions, 'positions should not be null').not.toBeNull();

    // Node IDs match written files
    const nodeIds = (snap.nodes as Array<{id: string}>).map(n => n.id);
    expect(nodeIds).toContain(fileA);
    expect(nodeIds).toContain(fileB);

    // Edges reflect real import (a → b)
    expect((snap.edges as unknown[]).length, 'Expected at least 1 edge from a.ts import').toBeGreaterThan(0);

    // Error case: 404 for missing snapshot ID
    const missingRes = await page.request.get(`${SERVER_URL}/api/snapshot/999999999`);
    expect(missingRes.status(), '404 for missing snapshot ID').toBe(404);

    // Error case: 400 for non-numeric ID
    const invalidRes = await page.request.get(`${SERVER_URL}/api/snapshot/not-a-number`);
    expect(invalidRes.status(), '400 for non-numeric snapshot ID').toBe(400);
  } finally {
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```

### Example 4: Concurrency test

```typescript
test('Writing files during an active session does not cause the pipeline to pause...', async ({ page }) => {
  test.setTimeout(60000);
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'archlens-ph15-conc-'));
  try {
    // Write initial files and switch watch root
    for (let i = 0; i < 6; i++) {
      fs.writeFileSync(path.join(tmpDir, `init${i}.ts`), `export const init${i} = ${i};\n`);
    }
    await page.request.post(`${SERVER_URL}/api/watch`, { data: { directory: tmpDir } });

    // Wait for initial snapshot
    const initDeadline = Date.now() + 30000;
    while (Date.now() < initDeadline) {
      await page.waitForTimeout(1000);
      const res = await page.request.get(`${SERVER_URL}/api/timeline`);
      if (res.status() === 200) {
        const list = (await res.json()) as unknown[];
        if (list.length > 0) break;
      }
    }

    // Phase: write 10+ files to trigger new snapshot threshold while simultaneously reading
    const writeStart = Date.now();
    for (let i = 0; i < 12; i++) {
      fs.writeFileSync(path.join(tmpDir, `concurrent${i}.ts`), `export const c${i} = ${i};\n`);
    }

    // Concurrent read — must respond within 5s even during write burst
    const pollRes = await page.request.get(`${SERVER_URL}/api/timeline`);
    const pollEnd = Date.now();
    expect(pollRes.status(), 'GET /api/timeline should return 200 during concurrent writes').toBe(200);
    expect(pollEnd - writeStart, 'Timeline response should arrive within 5s during write burst').toBeLessThan(5000);

    // After burst, poll to confirm new snapshots appear (proves pipeline didn't drop events)
    const postBurstDeadline = Date.now() + 30000;
    let finalSnapshots: unknown[] = [];
    while (Date.now() < postBurstDeadline) {
      await page.waitForTimeout(1000);
      const res = await page.request.get(`${SERVER_URL}/api/timeline`);
      if (res.status() === 200) {
        finalSnapshots = (await res.json()) as unknown[];
        if (finalSnapshots.length > 1) break; // More than initial snapshot
      }
    }
    // We should see more than the initial snapshot (pipeline is still processing)
    expect(finalSnapshots.length, 'Expected new snapshots after concurrent writes').toBeGreaterThan(0);
  } finally {
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```
