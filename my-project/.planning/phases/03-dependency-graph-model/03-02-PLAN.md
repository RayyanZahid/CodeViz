**1. Create `packages/server/src/graph/GraphPersistence.ts`:**

This module provides two functions: `persistDelta` (write-through) and `loadGraphState` (startup load).

**`persistDelta(delta: GraphDelta, getNodeMetadata: (id: string) => NodeMetadata | undefined): void`**
- Wraps all mutations in a single `db.transaction((tx) => { ... })` call for atomicity
- Inside the transaction:
  - For each node in `delta.addedNodes` and `delta.modifiedNodes`: get NodeMetadata from the graph via the callback, convert to a GraphNodeInsert row, call `tx.insert(graphNodes).values(row).onConflictDoUpdate(...)`. Map NodeMetadata fields to DB columns:
    - `id` = nodeId (filePath)
    - `name` = last path segment (filename without extension)
    - `nodeType` = NodeType.SERVICE_MODULE (default — inference engine in Phase 4 will reclassify)
    - `zone` = null (Phase 4 assigns zones)
    - `fileList` = [nodeId] (single file per node for now)
    - `incomingEdgeCount` / `outgoingEdgeCount` = count from the graphlib instance (pass as additional arg or compute from delta)
    - `lastModified` = new Date(metadata.lastModified)
    - `createdAt` = new Date() (only on insert, not on conflict update)
  - For each nodeId in `delta.removedNodeIds`: delete edges first via `tx.delete(graphEdges).where(or(eq(graphEdges.sourceId, nodeId), eq(graphEdges.targetId, nodeId))).run()`, then delete node via `tx.delete(graphNodes).where(eq(graphNodes.id, nodeId)).run()`
  - For each edge in `delta.addedEdges`: insert via `tx.insert(graphEdges).values({ id: \`${edge.v}->${edge.w}\`, sourceId: edge.v, targetId: edge.w, edgeType: EdgeType.IMPORTS_DEPENDS_ON, createdAt: new Date() }).onConflictDoUpdate(...)` (upsert in case edge ID already exists from a previous session)
  - For each edgeId in `delta.removedEdgeIds`: `tx.delete(graphEdges).where(eq(graphEdges.id, edgeId)).run()`
- Import `db` from `../db/connection.js`, schema from `../db/schema.js`, types from `drizzle-orm`
- Import `NodeType, EdgeType` from `@archlens/shared/types`
- All operations are synchronous (better-sqlite3 is sync, Drizzle wraps it synchronously)

**`loadGraphState(): { nodes: Array<{ id: string; metadata: NodeMetadata }>; edges: Array<{ source: string; target: string; symbols: string[] }> }`**
- Query all nodes via `db.select().from(graphNodes).all()`
- Query all edges via `db.select().from(graphEdges).all()`
- Map DB rows back to NodeMetadata and edge data
- For edges, parse the `id` field to extract symbols (or store symbols in a column — but the current schema doesn't have a symbols column, so store symbols as JSON in the edgeType column or use the existing edge structure). Actually, the existing graphEdges schema has `edgeType: text` — this is a type discriminator, not a place for symbols. Since the DB schema doesn't have a symbols column, for Phase 3 load edges from DB without symbol data (symbols will be repopulated when files are re-parsed on startup via chokidar ignoreInitial:false). The edge row provides structural connectivity; symbols are runtime-only in the in-memory graph.
- Return the structured data for DependencyGraph to consume

3. After Fastify server starts listening, add the graph + pipeline initialization:

```typescript
// Initialize dependency graph
const graph = new DependencyGraph();
graph.loadFromDatabase();

// Log delta events for visibility
graph.on('delta', (delta) => {
  console.log(`[Graph] Delta v${delta.version}: +${delta.addedNodes.length} nodes, -${delta.removedNodeIds.length} nodes, ~${delta.modifiedNodes.length} modified, +${delta.addedEdges.length} edges, -${delta.removedEdgeIds.length} edges`);
  if (delta.cyclesAdded.length > 0) {
    console.log(`[Graph] Cycles detected: ${delta.cyclesAdded.length} new`);
  }
});

// Start pipeline — watches for file changes, parses, feeds into graph
const watchRoot = process.env.ARCHLENS_WATCH_ROOT || process.cwd();
const pipeline = new Pipeline(
  watchRoot,
  (batch) => graph.onParseResult(batch),
);
await pipeline.start();
console.log(`[ArchLens] Watching ${watchRoot} for changes`);
```

4. Add cleanup on server close: before process exit, call `pipeline.stop()`.

**Important notes:**
- The Pipeline constructor's second argument is the `onResult` callback — pass `(batch) => graph.onParseResult(batch)` (arrow function to preserve `this` context)
- `graph.loadFromDatabase()` must be called BEFORE `pipeline.start()` so prior state is loaded before chokidar emits initial file events
- Use `ARCHLENS_WATCH_ROOT` env var for the directory to watch — defaults to cwd for development
- Keep the existing health endpoint and Fastify setup intact — only add graph + pipeline initialization after the server is listening
- Do NOT import Pipeline at the top if it's already imported — check first
