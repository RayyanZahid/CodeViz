  <name>Task 1: Implement ConfigLoader for .archlens.json zone overrides</name>
  <files>packages/server/src/inference/ConfigLoader.ts</files>
  <action>
Create `packages/server/src/inference/ConfigLoader.ts` that loads and watches a `.archlens.json` configuration file from the project's watch root directory.

**ArchLensConfig interface:**
```typescript
interface ArchLensConfig {
  zoneOverrides?: Record<string, ZoneName>; // key: filePath or glob pattern, value: zone name
}
```

**ConfigLoader class:**
- Constructor takes `watchRoot: string` (the ARCHLENS_WATCH_ROOT used by Pipeline).
- `configPath` = `path.join(watchRoot, '.archlens.json')`.
- On construction, call `loadConfig()` to read the file synchronously.
- Use `fs.readFileSync(configPath, 'utf-8')` wrapped in try/catch — if file doesn't exist or is invalid JSON, use empty config `{}`.
- Set up a chokidar watcher (reuse the chokidar import pattern from Phase 2) on the single config file path. On 'change' or 'add' events, call `loadConfig()` again. On 'unlink', reset config to `{}`.

**getOverride(nodeId: string): ZoneName | null:**
1. If no zoneOverrides in config, return null.
2. **Exact match first:** Check `overrides[nodeId]` directly.
3. **Glob match second:** Iterate over override entries and use `micromatch.isMatch(nodeId, pattern)` for each. Return the first matching zone.
4. Return null if no match.

**destroy():** Stop the chokidar watcher (for graceful shutdown).

Import micromatch with: `import micromatch from 'micromatch';`
Import ZoneName from `@archlens/shared/types`.
Import chokidar from `chokidar`.
Import fs from `node:fs` and path from `node:path`.

**Key considerations (from RESEARCH.md):**
- Config file may not exist — that's normal, not an error.
- Override keys can be exact paths ("src/utils/format.ts") or globs ("src/lib/**/*.ts").
- Override zone values must be valid ZoneName values.

<task type="auto">
  <name>Task 2: Implement ZoneClassifier with path-first, topology-second classification</name>
  <files>packages/server/src/inference/ZoneClassifier.ts</files>
  <action>
Create `packages/server/src/inference/ZoneClassifier.ts` implementing the two-pass zone classification strategy.

**Exported ZONE_PATH_PATTERNS** — ordered array of `{ zone: ZoneName; patterns: RegExp[] }`:
- **frontend:** `/\/(components?|pages?|views?|ui|screens?|layouts?)\//i`, `/\/(hooks?|contexts?)\//i`, `/\.(tsx|jsx)$/i`
- **api:** `/\/(routes?|controllers?|handlers?|endpoints?|api)\//i`, `/\/(middleware|middlewares?)\//i`, `/\.(routes?|controller|handler)\.[tj]s$/i`
- **services:** `/\/(services?|use-?cases?|business|domain|application)\//i`, `/\.service\.[tj]s$/i`, `/\/(managers?|processors?|orchestrators?)\//i`
- **data-stores:** `/\/(repositories?|models?|entities?|schemas?|migrations?|db|database)\//i`, `/\.(repository|model|entity|schema|migration)\.[tj]s$/i`, `/\/(prisma|drizzle|typeorm|sequelize)\//i`
- **infrastructure:** `/\/(config|configs?|infra|infrastructure|setup|bootstrap|server)\//i`, `/\/(workers?|jobs?|queues?|tasks?|cron)\//i`, `/\/(plugins?|adapters?|connectors?)\//i`, `/\/(utils?|helpers?|lib|libs?|shared)\//i`
- **external:** `/^__ext__\//` (external stub nodes from Phase 3)

**ZoneClassifier class:**
- Constructor takes `configLoader: ConfigLoader` and `graph: DependencyGraph` (from Plan 01 accessor methods).
- Maintains an in-memory `zoneCache: Map<string, ZoneName>` for quick lookups during topology classification.

**classify(nodeId: string): ZoneName:**
1. Check `configLoader.getOverride(nodeId)` first — if override exists, return it (ARCH-06: user override takes precedence).
2. Call `classifyByPath(nodeId)` — if result is not 'unknown', return it. (Path wins per locked decision: "When path says API but imports say frontend, path wins the tie.")
3. Call `classifyByTopology(nodeId)` — return result or 'unknown'.

**classifyByPath(nodeId: string): ZoneName** — iterate ZONE_PATH_PATTERNS, return first zone where any pattern matches. Return 'unknown' if none match.

**classifyByTopology(nodeId: string): ZoneName | null** — majority-vote over neighbors:
1. Get predecessors + successors from `this.graph.getPredecessors(nodeId)` and `this.graph.getSuccessors(nodeId)`.
2. Filter out `__ext__/` nodes.
3. For each neighbor, look up zone from `this.zoneCache` (NOT from SQLite — use in-memory cache from current delta processing).
4. Count zones (exclude 'unknown'). If no non-unknown neighbors, return null.
5. Return the zone with the most neighbors (majority vote).

**classifyDelta(delta: GraphDelta): ZoneUpdate[]** — the main entry point called by InferenceEngine:
1. **Pass 1 — Path classification:** For each node in `delta.addedNodes` and `delta.modifiedNodes`, call `classify(nodeId)`. Collect ZoneUpdate objects (with previousZone from zoneCache). Update zoneCache immediately so Pass 2 can use the new classifications.
2. **Pass 2 — Unknown re-evaluation:** After Pass 1, collect all nodes in zoneCache still marked 'unknown'. For each, check if any neighbor was just classified to non-unknown in Pass 1. If so, re-classify by topology. This implements the locked decision: "Unclassifiable files start in unknown but are re-evaluated when more graph data arrives."
3. Return all ZoneUpdate objects (from both passes) where zone actually changed.

**Important anti-patterns to avoid (from RESEARCH.md):**
- Do NOT re-classify the entire graph on every delta — only classify nodes in addedNodes and modifiedNodes plus unknown neighbors.
- Do NOT use framework-specific detection (no Next.js, Express, FastAPI recognition) per locked decisions.
- Two-pass processing within a single delta prevents Pitfall 4 (stale zone data during initial burst).

**updateZoneCache(nodeId: string, zone: ZoneName):** Public method to allow InferenceEngine to seed the cache from SQLite-loaded zones on startup.
  </action>
