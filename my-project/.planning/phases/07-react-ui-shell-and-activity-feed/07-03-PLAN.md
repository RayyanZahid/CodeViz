  <action>
    **1. Add `panToNode(worldX: number, worldY: number)` to `packages/client/src/canvas/ViewportController.ts`:**

    This method centers the viewport on a given world-space coordinate. Read the existing `fitToView()` implementation for the pattern — it already manipulates `stage.position()` and `stage.scale()`.

    Implementation:
    ```typescript
    panToNode(worldX: number, worldY: number): void {
      const stage = this.stage;
      const scale = stage.scaleX();
      // Center the given world point in the viewport
      const newX = stage.width() / 2 - worldX * scale;
      const newY = stage.height() / 2 - worldY * scale;
      stage.position({ x: newX, y: newY });
      stage.batchDraw();
      this.persistViewport();
      this.onViewportChange?.();
    }
    ```

    Ensure `persistViewport()` and `onViewportChange` are already called in similar methods (they are in `fitToView`). If `onViewportChange` is a different name, match the existing callback pattern.

    **2. Expose node world positions from `packages/client/src/canvas/ArchCanvas.tsx`:**

    The ArchCanvas useEffect has access to `nodeRenderer` which stores node positions. Add an `onNodePositionQuery` pattern: store a `nodePositionsRef` (a `Map<string, {x: number, y: number}>`) at module level or via a ref. Update it inside the `graphStore.subscribe()` callback after `placer.placeNewNodes()` runs — copy positions from the placer.

    Simpler approach: Add a new prop to ArchCanvas:
    ```typescript
    nodePositionsRef?: React.MutableRefObject<Map<string, {x: number, y: number}>>;
    ```

    Inside the subscribe callback, after `placer.placeNewNodes()`, update `nodePositionsRef.current` with the latest positions from `placer.getPositions()`.

    This allows App.tsx to read node world positions for the cross-panel navigation callback without ArchCanvas needing to know about panels.

    **3. Expose imperative `selectNodeOnCanvas(nodeId: string)` method from ArchCanvas (per locked user decision: "Selecting a node highlights its dependency edges on the canvas"):**

    The existing canvas click handler calls `clearSelection()`, `highlightNode()`, and `highlightDependency()` for outgoing edges — but this only fires on direct canvas clicks. When the sidebar sets `selectedNodeId` (e.g., via `handleHighlightNode`), the canvas does NOT highlight edges. To fix this:

    Add a new prop to ArchCanvas:
    ```typescript
    canvasRef?: React.MutableRefObject<{ selectNodeOnCanvas: (nodeId: string) => void } | null>;
    ```

    Inside the main `useEffect` (after `nodeRenderer` and the click handler are set up), assign the imperative handle:
    ```typescript
    if (canvasRef) {
      canvasRef.current = {
        selectNodeOnCanvas: (nodeId: string) => {
          clearSelection(nodeRenderer);
          highlightNode(nodeId, nodeRenderer);
          // Highlight dependency edges (same logic as click handler)
          const currentState = graphStore.getState();
          for (const edge of currentState.edges.values()) {
            if (edge.sourceId === nodeId) {
              highlightDependency(edge.targetId, nodeRenderer);
            }
          }
          handleSelectNodeRef.current(nodeId);
          gl.batchDraw();
        },
      };
    }
    ```

    In the cleanup return, set `canvasRef.current = null`.

    Also add `canvasRef` to the ArchCanvasProps interface alongside the existing `viewportControllerRef` and `nodePositionsRef` props.
