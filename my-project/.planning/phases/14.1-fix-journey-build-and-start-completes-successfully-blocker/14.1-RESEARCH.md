## Summary

This phase fixes two independent failures that prevent the "Build and Start" journey test from passing. First, the root `package.json` has no `build` script (only `dev`, `build:workers`, and `typecheck`). The journey test calls `npm run build` which fails immediately. Second, the Playwright config has no `webServer` block, so the "Dev server starts and responds" and "No JavaScript errors on load" tests depend on an externally running server — making them unreliable in CI.

The fix is narrow and well-understood: add a `build` script to root `package.json` that runs `tsc -b` for all packages, fix the journey test to use `pnpm build`, add dist-folder verification, and add a `webServer` block to the Playwright config that runs `pnpm dev` with a 60-second timeout. The primary technical challenge is that the client's `tsconfig.json` currently has `noEmit: true`, which must be addressed to allow the build to produce actual output. The server and shared packages need `composite: true` set for `tsc -b` project references to work correctly.

**Primary recommendation:** Add root `build` script as `pnpm build:workers && tsc -b packages/shared packages/server packages/client`, fix client tsconfig to emit to `dist/`, add `composite: true` to all three package tsconfigs, update the journey test to use `pnpm build` and verify dist output, and add the Playwright `webServer` block.

### Pattern 1: tsc -b (TypeScript Build Mode with Project References)

**What:** `tsc -b` builds multiple tsconfig files in dependency order. Each tsconfig must have `composite: true` for referenced configs to be eligible as project references.

**When to use:** Multi-package monorepos where TypeScript packages depend on each other. Faster than independent builds — only rebuilds what changed (tracked via `.tsbuildinfo` files).

**Requirements for each tsconfig participating in `-b`:**
```json
{
  "compilerOptions": {
    "composite": true,       // REQUIRED — enables incremental, produces .d.ts
    "declaration": true,     // REQUIRED — produces .d.ts for referencing packages
    "declarationMap": true   // Optional but recommended
  }
}
```

The root `tsconfig.base.json` already sets `"declaration": true` and `"declarationMap": true`. Adding `"composite": true` to each package tsconfig is the primary missing piece.

**Note on shared package:** The shared package has `.js` files alongside `.ts` files in `src/types/` (already emitted). This means shared may already be in a state where it can be treated as pre-compiled. However, to be safe and consistent, adding `composite: true` + `outDir: dist` is cleaner.

**Client-specific consideration:** Client's `tsconfig.json` has `noEmit: true`, which directly conflicts with `tsc -b` producing output. For the build script to generate verifiable dist output (per the locked decision), `noEmit` must be removed or set to `false` and `outDir: dist` must be added. This is safe because the root `typecheck` script already runs `tsc -b` on server+client — but if typecheck was relying on `noEmit` to skip output, it now produces output too.

**Alternative approach (separate build tsconfig for client):** Create `packages/client/tsconfig.build.json` that extends the base but removes `noEmit` and adds `outDir: dist`. The root `build` script references this instead of `packages/client`. This avoids modifying the existing client tsconfig that Vite relies on. This is cleaner and worth considering as Claude's discretion.

### Pattern 2: Playwright webServer Block

**What:** Playwright's `webServer` config option instructs the test runner to start a process before running tests and stop it after. It polls a URL to detect readiness.

**Official pattern (from Playwright docs):**
```typescript
// Source: Playwright official docs — defineConfig webServer
import { defineConfig } from '@playwright/test';

export default defineConfig({
  webServer: {
    command: 'pnpm dev',
    url: 'http://localhost:5173',
    timeout: 60 * 1000,          // 60 seconds
    reuseExistingServer: !process.env.CI,  // reuse in local dev, fresh in CI
    cwd: '/path/to/project/root',
  },
  // ... rest of config
});
```

**Key parameters:**
- `command`: The shell command to start the server. For this project: `pnpm dev` (starts both Fastify server on 3100 and Vite client on 5173 via concurrently).
- `url`: The URL Playwright polls to detect readiness. Should be the client URL `http://localhost:5173` since that's what tests navigate to.
- `timeout`: Milliseconds to wait for readiness. 60000 (60s) is appropriate given first-run TypeScript compilation and tsx startup.
- `reuseExistingServer`: If `true` and a server is already listening at the URL, Playwright skips starting a new one. Useful for local dev where the dev server is already running.
- `cwd`: Working directory for the command. Must point to the root of the project where `pnpm dev` should be run.

**The `cwd` problem:** The Playwright config is at `.auto-gsd/journey-tests/playwright.config.ts`. The `pnpm dev` command needs to run from the project root, not from `.auto-gsd/journey-tests/`. The `cwd` option must be set to `../../` (relative) or the absolute project root path. Using `path.resolve(__dirname, '../../')` in the config is the standard pattern.

### Pattern 3: Journey Test Corrections

**Current test (broken):**

**Fixed test:**

```typescript
expect(serverJsFiles.length).toBeGreaterThan(0);
```

**Note on `cwd` in execSync:** The journey test currently runs `npm run build` without a `cwd` option, meaning it runs from whatever directory the test runner's cwd is. This likely fails silently or uses the wrong directory. The corrected test should explicitly set `cwd` to the project root.

### Anti-Patterns to Avoid

- **`npm run build` in a pnpm project:** npm's `run` command may fail in a pnpm workspace if npm's scripts don't understand pnpm workspace protocol. Always use `pnpm` in pnpm projects.
- **Missing `composite: true` in `tsc -b`:** Without `composite`, TypeScript throws `error TS6306: Referenced project must have setting "composite": true`. This is a hard blocker.
- **`noEmit: true` with `tsc -b` and output verification:** If `noEmit: true` remains on any package, `tsc -b` will typecheck that package but produce no `.js` files. The dist folder check will fail.
- **Playwright `webServer` without `cwd`:** If the config is in a subdirectory and `cwd` is not set, `pnpm dev` runs from the wrong directory and fails to find the workspace `pnpm-workspace.yaml`.
- **30-second timeout for build:** `tsc -b` on a multi-package TypeScript project (with workers build included) can easily take 30+ seconds on first run. The journey test's `timeout: 30000` will need to increase to at least 120000 for the build test.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Running server before tests | Custom test setup/teardown scripts | Playwright `webServer` config | Built-in, handles process lifecycle, polls readiness, tears down after suite |
| Cross-platform path resolution in test | String concatenation for paths | `path.resolve()` + `import.meta.url` | Handles Windows/Unix differences; already an ESM project |
| Detecting server readiness | Manual polling loops | Playwright `webServer.url` poll | Built into Playwright; exponential backoff, proper timeout handling |

**Key insight:** Playwright's `webServer` option is purpose-built for this exact pattern — start a dev server, wait for it to be ready, run tests, tear down. Custom shell scripts or test setup/teardown would duplicate this poorly.

## Common Pitfalls

### Pitfall 1: `composite: true` required for all tsc -b participants

**What goes wrong:** `error TS6306: Referenced project 'packages/shared' must have setting "composite": true` — build fails immediately.

**Why it happens:** `tsc -b` requires each tsconfig referenced in the build to opt into composite mode. The current tsconfigs do not have `composite: true`.

**How to avoid:** Add `composite: true` to `packages/shared/tsconfig.json`, `packages/server/tsconfig.json`, and `packages/client/tsconfig.json` (or the build variant of client's tsconfig).

**Warning signs:** TypeScript error TS6306 in console output when running `tsc -b`.

### Pitfall 2: Client `noEmit: true` blocks dist verification

**What goes wrong:** `tsc -b` completes without error but `packages/client/dist/` never gets created, causing the dist folder verification check to fail.

**Why it happens:** `noEmit: true` is explicitly set in `packages/client/tsconfig.json`. This is correct for Vite dev mode (Vite handles its own bundling), but breaks the TypeScript-only build that needs to produce output.

**How to avoid:** Either remove `noEmit: true` and add `outDir: dist` to the client tsconfig, or create a separate `tsconfig.build.json` for the client that overrides these options and reference that in the root `build` script.

**Warning signs:** `packages/client/dist/` does not exist after `tsc -b` runs.

### Pitfall 3: `cwd` mismatch for execSync and webServer

**What goes wrong:** `execSync('pnpm build')` runs from the `.auto-gsd/journey-tests/` directory instead of the project root, causing "No pnpm workspace found" or "script not found" errors.

**Why it happens:** Node.js `execSync` inherits the process cwd, which for Playwright tests is typically the playwright config directory.

**How to avoid:** Always pass `cwd` explicitly to `execSync`. Use `path.resolve(new URL(import.meta.url).pathname, '../../..')` (three levels up: journey-tests → .auto-gsd → my-project root) to compute the project root from ESM `import.meta.url`.

**Warning signs:** "No pnpm-workspace.yaml found" or "ERR_PNPM_NO_SCRIPT" errors in test output.

### Pitfall 4: Playwright `webServer` timeout too short

**What goes wrong:** Playwright times out waiting for `http://localhost:5173` to respond before `pnpm dev` has finished starting tsx and Vite.

**Why it happens:** Default timeout may be too short; first-run compilation with tsx takes 5-15 seconds, Vite starts in 1-3 seconds, but `pnpm dev` also runs `build:workers` first.

**How to avoid:** Set `timeout: 60000` (60 seconds) in the `webServer` block. The CONTEXT.md specifies this explicitly.

**Warning signs:** "Timed out waiting for http://localhost:5173 to be available" in test output.

### Pitfall 5: `build:workers` must run before `tsc -b`

**What goes wrong:** `tsc -b` compiles worker files that import from `dist/` outputs of the worker build, causing import errors if workers aren't compiled first.

**Why it happens:** The server's `build:workers` produces CJS files for the Piscina worker pool that the main server code references.

**How to avoid:** Root `build` script should run `pnpm build:workers && tsc -b ...` (workers first, then full TypeScript build). This mirrors the `dev` script pattern which also runs `pnpm build:workers` first.

**Warning signs:** TypeScript errors about missing worker module files during `tsc -b`.

### Pitfall 6: Shared package `.js` files in src/ conflict with outDir

**What goes wrong:** The shared package already has `.js` files alongside `.ts` files in `src/types/` (committed to the repo). If `outDir: dist` is added, `tsc -b` would produce a separate `dist/` but the package exports still point to `src/types/index.ts`.

**Why it happens:** The shared package's `package.json` exports use `"./types": "./src/types/index.ts"` — TypeScript source paths, not compiled output. This works in workspace context via TypeScript's path resolution.

**How to avoid:** Two options:
1. Leave shared as-is (no outDir, just add `composite: true`). For `tsc -b`, shared participates as a typecheck target but doesn't need to emit separately since the workspace resolves its source TypeScript directly.
2. Add `outDir: dist` and update `package.json` exports accordingly.

Option 1 is simpler and lower risk. The dist verification only needs to check `packages/server/dist` and `packages/client/dist`.

## Open Questions

1. **`reuseExistingServer` setting for webServer**
   - What we know: `true` allows reusing an already-running server (good for local dev); `false` or `!process.env.CI` starts fresh in CI
   - What's unclear: The gsd runner's CI environment — does it run tests in an environment where a server would already be running?
   - Recommendation: Default to `reuseExistingServer: true` for now. This is the safest choice: if a dev server is already running (common in local dev), tests use it; if not, Playwright starts one. In CI, no server would be pre-running so it starts fresh regardless.

2. **Client tsconfig approach: modify existing vs. create tsconfig.build.json**
   - What we know: Option A (modify) is simpler; Option B (separate) preserves existing behavior for Vite dev exactly
   - What's unclear: Whether Vite uses `tsconfig.json` for anything other than type checking (Vite uses `esbuild` internally for transpilation, so it generally doesn't care about `noEmit` or `outDir`)
   - Recommendation: Use Option A (modify existing tsconfig.json). Vite does not use `outDir` or `noEmit` — these are TypeScript compiler options that Vite ignores. Removing `noEmit` and adding `outDir: dist` has no effect on the Vite dev server or HMR. This is simpler and avoids the extra file.

3. **Shared package emit strategy**
   - What we know: Shared's `package.json` exports point directly to `.ts` source (`"./types": "./src/types/index.ts"`). The `src/types/` directory already contains both `.ts` and `.js` files (pre-emitted).
   - What's unclear: Whether those `.js` files in src/ were intentionally committed or are artifacts.
   - Recommendation: Add `composite: true` to shared's tsconfig (required for `tsc -b`) but do NOT add `outDir`. Leave the existing exports as-is. The dist verification check only needs to cover server and client.
