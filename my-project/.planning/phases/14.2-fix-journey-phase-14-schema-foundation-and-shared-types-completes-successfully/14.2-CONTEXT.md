<decisions>
## Implementation Decisions

### Test verification strategy
- Use REST API diagnostic endpoints to bridge browser-based Playwright tests to server-side behavior
- Diagnostic endpoints are dev/test only — guarded behind NODE_ENV check, not available in production
- TypeScript compilation verification: if the server is running and responding, shared types compiled successfully (no explicit tsc check needed)

### API surface for testing
- Separate diagnostic endpoints per success criterion (not one combined endpoint) — enables granular failure reporting in tests
- Endpoints perform full CRUD proof: insert test row, read it back, delete it — proves read/write capability
- Auto-cleanup: test data is inserted, verified, and deleted in one request — no residue
- Response shape: `{ok: true, table: "graph_snapshots", columns: [...], crud: {insert: true, read: true, delete: true}}` — pass/fail with details

### Delta threshold testing
- Use real file writes to trigger the watcher pipeline (not simulated events)
- Point watcher at a temp directory — gets wiped after test, no cleanup needed in real project
- Assertion strategy: write 5-10 files, query snapshot count via API, assert `snapshot_count < file_change_count`

### Test scope boundaries
- All 4 tests remain as Playwright tests — use `page.request` to call diagnostic REST endpoints
- Verify `graphJson` column (which includes positions inside JSON payload) rather than adding a literal `positions_json` column — match actual schema, not literal criteria wording
- Custom per-test timeouts: schema/CRUD tests get short timeouts (~10s), delta threshold test gets longer timeout (30-60s) for file processing pipeline

### Claude's Discretion
- Route prefix for diagnostic endpoints (e.g. `/api/debug/*` vs `/api/health/*`)
- Exact temp directory location and creation/teardown strategy
- Number of test files within the 5-10 range
- Wait/polling strategy for delta threshold test

</decisions>
