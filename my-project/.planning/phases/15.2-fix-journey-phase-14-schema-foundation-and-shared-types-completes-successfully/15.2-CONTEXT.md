- Trigger 15-20 real file changes in the watched directory to generate events
- Use /api/timeline to check snapshot count after changes settle

- Use existing diagnostic endpoints (/api/debug/graph-snapshots-table, /api/debug/intent-sessions-table) for CRUD tests — they already perform insert/read/delete internally

- No cleanup needed — diagnostic endpoints self-clean, temp files are harmless in test env

### Pass/fail criteria
- graph_snapshots test: passes if diagnostic endpoint returns `ok: true`
- intent_sessions test: passes if diagnostic endpoint returns `ok: true`
- shared types test: passes if /api/debug/shared-types returns `ok: true` AND specific type names are available
- delta threshold test: after 15-20 file changes, snapshot count < file change count
- Include descriptive custom failure messages for all assertions (e.g., "Expected snapshot count < event count but got X snapshots for Y events")
- Set test timeout to 60s per test (up from 30s) for safety with debounce timing
