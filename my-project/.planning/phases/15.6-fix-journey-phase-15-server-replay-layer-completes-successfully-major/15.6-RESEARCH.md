## Summary

This is the 5th occurrence of the auto-gsd overwrite pattern. The OttoGSD `journey-test-generator.ts` module runs after EVERY phase completion and regenerates all `journey-*.spec.ts` files and `playwright.config.ts` from scratch with placeholder stubs. The current working tree has overwritten stubs in all 4 files, while git HEAD (commit `41f31a2`) already contains the correct versions for all 4 files.

This phase has two parts: (1) restore 4 files and (2) add a new `journey-canary.spec.ts` file that actively detects future overwrite events by scanning all spec files for placeholder patterns. The canary is the key new element distinguishing this phase from prior restore-only phases. The canary spec must be created fresh (not restored from git, since it doesn't exist yet) and must itself survive the auto-gsd overwrite pattern — since the canary is NOT a `journey-*.spec.ts` file and uses a name outside the `journey-` prefix deletion pattern... actually, since canary IS named `journey-canary.spec.ts`, it will be deleted by the generator. The canary's value is in immediately making overwrites visible in test results when they next occur.

**Primary recommendation:** Restore all 4 files from git HEAD using `git checkout HEAD -- <paths>`, create `journey-canary.spec.ts` as a new file that reads its sibling spec files and fails if any contain placeholder patterns, commit all 5 files, then run `npx playwright test` and confirm 13/13 pass.

### Pattern 2: Canary Test — Placeholder Detection via fs.readFileSync

**What:** A standalone Playwright test that uses Node.js `fs.readFileSync` to scan sibling spec files for placeholder patterns, then fails the test if any are found.

**When to use:** Created once and committed; auto-detected by Playwright since it matches `*.spec.ts` pattern.

**Detection patterns to catch:**
1. `bodyText.length` pattern — stub tests that only check body text length
2. `page.goto` + `body` check WITHOUT any `page.request` or `fetch` calls — generic browser-only stubs that don't hit real endpoints

**Example:**
```typescript
// journey-canary.spec.ts
import { test, expect } from '@playwright/test';
import fs from 'fs';
import path from 'path';
import { fileURLToPath } from 'url';

const __dirname = path.dirname(fileURLToPath(import.meta.url));

test('No spec files contain placeholder overwrite patterns', async () => {
  const specFiles = fs.readdirSync(__dirname)
    .filter(f => f.endsWith('.spec.ts') && f !== 'journey-canary.spec.ts');

  const violations: string[] = [];

  for (const filename of specFiles) {
    const content = fs.readFileSync(path.join(__dirname, filename), 'utf-8');

    // Pattern 1: bodyText.length check (hallmark of auto-gsd stub generation)
    if (content.includes('bodyText.length')) {
      violations.push(`${filename}: contains bodyText.length placeholder check`);
    }

    // Pattern 2: page.goto without page.request (browser-only stub without API calls)
    // Real tests use SERVER_URL + page.request for API assertions
    const hasPageGoto = content.includes('page.goto(');
    const hasApiCall = content.includes('page.request.');
    const hasServerUrl = content.includes('SERVER_URL');
    if (hasPageGoto && !hasApiCall && !hasServerUrl && filename !== 'journey-build-and-start.spec.ts') {
      violations.push(`${filename}: has page.goto without page.request API calls (likely placeholder)`);
    }
  }

  expect(
    violations,
    `Placeholder patterns detected in spec files — auto-gsd may have overwritten real tests:\n${violations.join('\n')}`
  ).toHaveLength(0);
});
```

**Key design note:** `journey-build-and-start.spec.ts` is exempt from the "no page.goto without SERVER_URL" rule because it legitimately uses `BASE_URL` for browser navigation tests. All other spec files should use `SERVER_URL` + `page.request`.

### Anti-Patterns to Avoid

- **Restoring from specific commit hashes instead of HEAD:** HEAD already has the correct versions for all 4 files. CONTEXT.md mentions specific commits (e0632d2 for phase-15, 6ac8e89 for phase-14, 5b4aabb for build-and-start) but git log confirms HEAD contains the same content — use HEAD for simplicity.
- **Using `npm run build` in journey-build-and-start.spec.ts:** The correct command is `pnpm build` with `cwd: projectRoot` and a 180s timeout. The HEAD version already has this.
- **Parallel test execution:** `workers: 1` is mandatory — tests share server state via POST /api/watch. Removing it causes race conditions.
- **Missing `NODE_ENV=development` in playwright.config.ts:** Without this, diagnostic endpoints at `/api/debug/*` are not registered in the server, causing Phase 14 tests to fail.
- **Using BASE_URL (5173) for Phase 14/15 API tests:** Phase 14 and 15 spec files use `SERVER_URL` (port 3100). Only journey-build-and-start.spec.ts uses `BASE_URL` (port 5173) for Vite browser navigation.
- **Canary checking itself:** `journey-canary.spec.ts` should exclude itself from the scan (`filename !== 'journey-canary.spec.ts'`).

## Common Pitfalls

### Pitfall 1: Confirming HEAD Has the Correct Versions Before Restoring

**What goes wrong:** Assuming HEAD has stubs (matching the working tree) and trying to use specific older commit hashes.
**Why it happens:** The working tree was overwritten by auto-gsd, but the overwrite was NOT committed. The commit `41f31a2` (most recent HEAD) was created by the previous phase (Phase 15.5/15.6 context docs) and did not touch the spec files.
**How to avoid:** Run `git diff HEAD -- .auto-gsd/journey-tests/journey-phase-14.spec.ts | head -5` first. A diff showing removal of SERVER_URL confirms working tree has stubs but HEAD has real files.
**Warning signs:** `git show HEAD:my-project/.auto-gsd/journey-tests/journey-phase-14.spec.ts | head -3` shows `const SERVER_URL` = real file in HEAD.

### Pitfall 2: Wrong Git Command Syntax for Path Resolution

**What goes wrong:** `git show <hash>:.auto-gsd/journey-tests/...` fails with "path exists at my-project/..." hint.
**Why it happens:** The git repository root is `/f/Directory/FrontierIntelligenceHack`, not `my-project/`. `git show` requires paths relative to the repo root.
**How to avoid:** For `git show`, use `git show <hash>:my-project/.auto-gsd/journey-tests/...`. For `git checkout HEAD --`, running from inside `my-project/` works with relative paths like `.auto-gsd/journey-tests/...`.
**Warning signs:** Error `fatal: path '...' exists, but not '...'` with hint about `my-project/` prefix.

### Pitfall 3: Canary Test Failing Immediately on journey-build-and-start.spec.ts

**What goes wrong:** Canary incorrectly flags `journey-build-and-start.spec.ts` as a placeholder because it uses `page.goto` without `page.request`.
**Why it happens:** The build-and-start spec legitimately tests browser navigation (not API endpoints), so it uses `page.goto(BASE_URL + '/')` without `SERVER_URL` or `page.request`.
**How to avoid:** Add an explicit exemption in the canary: `if (filename !== 'journey-build-and-start.spec.ts')` before checking for the missing-API-call pattern. The `bodyText.length` check can still apply to all files.
**Warning signs:** Canary test fails even after correct restoration with message about journey-build-and-start.spec.ts.

### Pitfall 4: Canary Will Be Overwritten Too (By Design)

**What goes wrong:** False expectation that `journey-canary.spec.ts` will survive auto-gsd overwrites.
**Why it happens:** The CONTEXT.md says canary scans ALL spec files for future-proofing, but the canary itself is named `journey-canary.spec.ts` which matches the `journey-*` deletion pattern in OttoGSD's generator.
**How to avoid:** Accept this. The canary's value is making the NEXT overwrite event immediately visible as a test failure. When auto-gsd runs again, it will also overwrite the canary — but the RESTORED 4 files will be replaced with stubs, making tests fail. The canary makes this failure happen fast and clearly rather than silently.
**Warning signs:** No warning signs needed — this is intentional behavior per the established pattern.

### Pitfall 5: 13 Tests Expected, Not 12

**What goes wrong:** Counting tests and expecting 12 (the Phase 15.5 count), then getting confused when 13 run.
**Why it happens:** This phase ADDS `journey-canary.spec.ts` with 1 canary test. 12 original + 1 canary = 13 total.
**How to avoid:** Set the success criterion to `13 passed, 0 failed` not `12 passed`.
**Warning signs:** `npx playwright test` output shows 12 tests total instead of 13 — canary file was not created or not picked up.

**Root Cause Summary (confirmed in Phase 15.5 research):**

The overwrite source is `OttoGSD/src/orchestrator/journey-test-generator.ts`. This module:
1. Runs after EVERY phase completes (triggered by `phase:complete` event)
2. Deletes ALL files starting with `journey-` in `.auto-gsd/journey-tests/`
3. Regenerates them as placeholder stubs using `bodyText.length > 0` checks
4. Also regenerates `playwright.config.ts` with a minimal config lacking `workers: 1`, `webServer`, and `NODE_ENV: development`
5. Has no "skip if file already has real content" guard

Prevention requires changes to the OttoGSD binary (out of scope per STATE.md Phase 15.4 decision).
